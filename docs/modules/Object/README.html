Object

VaNillaにおけるいわゆるゲームオブジェクトを取り扱うための基底モジュールです。
VaNillaではゲーム内に存在するすべての実体は Object として扱われます。

Object には任意の ExpresserComponent を複数関連付けすることができます。
ExpresserComponent については こちらを参照してください。

APIリファレンス

詳細なAPIドキュメントはVaNilla Object APIリファレンスをご覧ください。

ExpresserComponentの関連付け方法

ExpresserComponent の関連付け方法として、下記の4種類を提供しています。

  • 任意のObject に対して DefaultExpresserComponent 属性を付ける
  • コンストラクタ引数の expresser_components 配列に ExpresserComponent を入れる
  • ObjectFactory.Instantiate での生成時の additional_components 配列に ExpresserComponent を入れる
  • ObjectActivator.Activate での生成時の additional_components 配列に ExpresserComponent を入れる

それぞれの関連付け方法の違いについては、ExpresserComponentを参照してください。

Object クラス

基底となるクラスです。 単体でインスタンス化はできないため、必ず本クラスを継承した上で具体的なクラスを作成する必要があります。 (例: Enemy、Player、など)

Object から継承したクラスはすべて下記の仮想関数を実装する必要があります。

SetUp

コンストラクタ相当の機能を実装します。
基本的に、クラスの持つメンバ変数および自動プロパティはすべてSetUp メソッド内で初期化するようにしてください。
Objectはデストラクトされないまま使いまわされる ケースが多いため、
そのようなケースではコンストラクタが呼ばれないまま動作し、以前の状態を引き継いだまま動作するという非常に危険な状態となります。

上記に反するケースではSetUpでメンバ変数などを初期化する必要はありませんが、そのようなケースはほぼ無いと思われるので
SetUpでの初期化は必須 と思って実装した方がいいです。

TearDown

デストラクタ相当の機能を実装します。

SetUp内やExecute内で動的確保したメモリや、アクセスしている資源は必ずTearDownで解放するようにしてください。
上にも記述したように、Objectは使いまわされる前提で実装をする必要があるため、
メモリやリソースを確保したままTearDownが終了した場合、次回の使い回しでの動作時に意図しない挙動となる可能性が高いです。

Execute

インスタンス化後、定期的に実行されるべき処理を実装します。 current_time として現在時刻が渡ってきます。

実装の指針として、事前に用意可能なリソースはすべて事前に用意しておく というのがVaNillaエンジンの指針のため、 基本的にExecute内ではメモリやリソースの確保は行わないようにしてください。

ObjectActivator

ジェネリック型で指定されたObjectのインスタンスを生成、およびObjectExecutorへの登録処理を自動で実施するユーティリティです。
使いどころとしては、ObjectFactoryに存在していないような、ゲーム中で少数のみ生成されるObjectを作る際に使います。
Object であればなんでも指定して生成・登録できるため、ObjectFactoryよりも使いやすいですが、
一方で生成・破棄のコストパフォーマンスは ObjectFactory に劣ります。
大量の Object を生成するケースでは ObjectFactory を利用し、少数部の生成が想定されるケースでは ObjectActivator の利用を推奨します。

ObjectActivator.Activate

Objectを継承する任意の型をジェネリック引数に指定して、生成およびObjectExecutorへの登録処理を実施します。
平たく言うと、ゲーム内にObjectを出現させます。
本APIで生成したObjectは ObjectActivator.Deactivate で破棄することを推奨します。

ObjectActivator.Deactivate

指定した Object をObjectExecutorから登録解除し、破棄します。
ObjectActivator.Activate とセットで運用することを想定しています。

ObjectExecutionPriorityAttribute

Object に付与する属性です。 Object の実行優先度を制御します。 ExpresserComponentExecutionOrder との実行順序制御の兼ね合いについて、 まず ExpresserComponentExecutionOrder で設定した優先度が優先されます。 その後、各優先度内での ObjectExecutionPriorityAttribute の値に応じて実行順序が決定されます。 本属性は付与したObjectに対して一律で同様に作用するため、特定のインスタンスのみ優先度を変更することはできません。

ObjectExecutionPriorityCollector

ObjectExecutionPriorityAttribute を管理するためのマネージャクラスです。 実行時（ゲーム起動時）に、存在するすべての ObjectExecutionPriorityAttribute を収集してキャッシュ化します。 本クラスの実装をVaNillaエンジンの使用者が詳細に理解する必要性は低いです。

ObjectTag

定義された優先度および Object の種別タグを enum 値で保持するファイルです。 本ファイルは Fixture/Script/process.bat を実行することで、ソースファイルレベルで事前生成されます。 詳細は下記 Fixture/Script/process.bat (convert.py)の項目を確認してください。

Fixture/Script/process.bat (convert.py)

上記 ObjectTag を事前生成するためのPythonスクリプトです。 Fixture/Script/object_list.yaml に記載されている項目を基にして、実行順序定義ファイルを事前生成します。 性質上、本batファイルを起動してオブジェクトタグ定義および実行優先度定義を生成しない場合、 ObjectExecutionPriorityAttribute 属性を使用しているファイルでコンパイルエラーとなる可能性が非常に高いです。 はじめに本batファイルを実行して ObjectTag を事前生成することを推奨します。 実行には Python3.6 以上の実行環境のほか、PyYAML パッケージのインストールが必要です。

VaNillaエンジン



VaNillaエンジンは、既存のゲームエンジン（DxLib・Unity）の上で動作する 「メタ・ゲームエンジン」 です。
VaNillaエンジンを利用することで、既存のゲームエンジンやゲームフレームワークの機能を使いつつも、
2Dアクションゲーム・2Dシューティングゲームなどの製作に特化した強力な機能を、
統一されたAPIで使うことができます。

✨ 主な特徴

⚡ 高性能オブジェクト処理 - 数千個のオブジェクトを軽量に処理
🧩 ExpresserComponent設計 - モジュラーで再利用可能なコンポーネントシステム
⚙️ ゼロコスト事前定義 - YAML設定と属性から、C#コードを事前生成
🏗️ 最新の.NETのパワーを活用 - 最新の.NET環境に追従
🎯 統一API - DxLibとUnityで同一のコードが動作(※開発中)
🎮 実戦実績 - ∀stralbringerなどの実際のゲーム開発で使用実績あり

🚀 クイックスタート

開発環境

  • Microsoft Visual Studio Community 2026 (64 ビット) Version 17.14.5 以降
  • .NET 10 SDK
  • Python 3.9+（コード生成用）
  • DxLib C# または Unity 2022.3+

セットアップ手順

  1. リポジトリをクローン

    git clone https://github.com/EndlessShirafu/VaNilla.git
cd VaNilla

  2. 設定ファイルから必要なコードを生成 ⚠️ 必須手順

    python VaNillaEngineManager/CodeGenerator.py -s [設定yamlファイルのパス]

  3. プロジェクトをビルド

    dotnet build

  4. テストプロジェクトを実行

    # DxLib版テストの実行
cd VaNillaTestProject/DxLib
dotnet run

⚠️ 重要: 手順2のコード生成を実行しないと、ObjectTagEnumやTextureResourceTypeが見つからないエラーでビルドが失敗します。

→続きは クイックスタートガイド を参照してください。

🏗️ アーキテクチャ概要

ExpresserComponent-Oriented Programming（表象要素指向プログラミング）

VaNillaエンジンの核となる設計思想です。
ゲームオブジェクトは「何を持っているか」ではなく 「何を表現するか」 で構成されます。

graph LR
   ID1[敵キャラクターA Object] --> ID2[敵キャラクターである]
   ID2 --> ID3[HPを持つ]
   ID2 --> ID4[画面描画される]
   ID4 --> ID5[座標を持つ]  
   ID2 --> ID6[回転できる]
   ID2 --> ID7[当たり判定がある]
   ID4 --> ID8[描画するテクスチャを持つ]  

   subgraph "ExpresserComponents"
   ID2
   ID3
   ID4
   ID5
   ID6
   ID7
   ID8
   end

「敵キャラクターである」ならば「HPを持つ」「画面に描画される」「当たり判定がある」
「画面に描画される」ならば「座標を持つ」「描画テクスチャを持つ」…といった形で、
コンポーネントの依存関係を属性で宣言することで、「敵キャラクターAの構築」に必要な機能が出揃います。

利点:

  • 📦 高い再利用性 - コンポーネントを組み合わせて新しいオブジェクトを作成
  • 🔧 保守性向上 - 機能が分離されており、個別に修正・テスト可能
  • 🧩 依存関係の明確化 - コンポーネント間の依存関係を属性で宣言

ゼロコスト事前定義システム

「事前に定義・計算可能な箇所は事前処理しておくことで、パフォーマンスを上げよう」という思想です。 具体的には、ゲームに関する設定はYAMLで記述し、ビルド前にC#コードを生成します。
これによって、リフレクションなどのいわゆる「C#の重い処理」を極限までカットしています。

graph LR
   ID1[設定YAML] --> ID2[CodeGenerator.py]
   ID2 --> ID3[C#コード生成]
   ID3 --> ID4[コンパイル]
   ID4 --> ID5[実行時オーバーヘッド0]

さらに、 Incremental Source Generator を活用することにより、 IDE上でのソースコードの補完を利かせつつも、事前処理によるパフォーマンスを担保することができます。

統一プラットフォーム抽象化

graph TB
   ID1[VaNilla API Layer] --> ID2[DxLib Implementation]
   ID1 --> ID3[Unity Implementation]
   ID2 --> ID4[DxLibDotNet]
   ID3 --> ID5[Unity Engine]

共通APIの下に Implementationレイヤー を設け、そこで各実装の差分を吸収します。
ユーザーは VaNilla API Layer のみを利用すればよいです。

Q&A

別ページへ分割予定です

  • Q. ビルドしたらObjectTagEnum が無いとか TextureResourceType が存在しないとかのエラーが出る
  • A. yaml設定ファイルからVaNillaエンジン用のC#設定ファイルが生成されていないのが原因です。 VaNillaEngineManager/CodeGenerator.exeで、設定用yamlファイルから設定を「生成」してから利用する必要があります。 詳しくはVaNillaEngineManager モジュールを参照してください。

  • Q. UnityのGameObjectと Object との間に関連性はあるの？

  • A. ありません。 ObjectはUnityのGameObjectと全く別の概念であり、独立に動作します。

  • 逆に、UnityのGameObjectに対して関連性を持たせたいならば、 Object の情報とUnityのGameObjectの情報を同期するような「HasRelationToUnityGameObject」的な ExpresserComponent を作るのがいいかもしれません。

  • 現状VaNillaエンジンでは上記のような、ゲームエンジン固有オブジェクトと Object との間で関連性を付けるモジュールは提供していません。

  • Q. Object を使うにはどうするの？
  • A. Object 自体は、 abstract なクラスのため、 そのままでは利用できません。 Object を継承した具体的なオブジェクトのクラスを自前で作成する必要があります。
    • 例えば、なんらかの「敵キャラクターA」というオブジェクトクラスを作りたい場合、 EnemyCharacter_A という Object を継承したクラスを作成します。
    • 詳しくは、 Object を参照してください。

  • Q. 上記に従い、「敵キャラクターA(EnemyCharacter_A)」という Object の具体実装を作ってみた。
    結局、これをどのようにして使えばよいの？

  • A. ObjectActivator を使うのが良いです。 ObjectActivator.Activate() を呼び出すことで、 ゲーム内に EnemyCharacter_A を出現させられます。

    • ObjectActivator.Activate() に EnemyCharacter_A をジェネリック型として指定することで、 EnemyCharacter_A のインスタンスを生成し、 ObjectExecutor に登録することができます。

    • ObjectActivator を介さずに普通に new すると、 EnemyCharacter_A のインスタンス自体は生成されますが、ゲーム内に「登録」はされません。 普通に new するケースでゲーム内に EnemyCharacter_A を「登録」するには、 ObjectExecutor.Register を別途呼び出す必要があります。

  • Q. 大量の（1000～10000個オーダーの）Objectを扱うにはどうすればいいの？

  • A. ObjectFactory を使うのが良いです。

    • ObjectFactory は、いわゆるオブジェクトプールです。 VaNillaエンジン初期化時に、指定された種別のObjectを事前に大量生成しておくことで、 大量のObjectの取り扱いを軽量に実施することができます。 また、ObjectFactory 経由で破棄した Object は ObjectFactory に格納されて「使いまわされる」ため、 破棄のコストがほぼ発生しません。 詳細は以下を参照してください。

  • ObjectFactory

  • ObjectPool

  • ObjectExecutor

  • Q. スレッドを使って実行処理を分割したい

  • A. 非常に強く非推奨です。 現状のVaNillaエンジンはシングルスレッドモデルで動作しており、マルチスレッドモデルによる動作を想定していません。

    • 無理矢理マルチスレッドで動作させた場合、リソースの競合などの発生によって容易にVaNillaエンジンの内部状態が不整合になったり、 例外が発生したりといった不具合現象が発生します。
    • たいていの場合、マルチスレッド動作によって実現したいことの1つには「パフォーマンスの向上」が挙げられますが、 パフォーマンスを向上させるためには、クリティカルセクションやロック機構によるリソースの保護、アフィニティマスクの適切な設定、プロセス優先度とスケジューリングポリシーの設定、コンテキストスイッチによるオーバーヘッドの勘案といった 様々な観点に基づいた 適切な設計および慎重な実装 が不可欠であり、正直VaNillaエンジン全体としては メンテ不能 に陥っていくことが予想されます。 マルチスレッド化によるパフォーマンス改善よりも、GCの回数を減らしたり、アルゴリズムレベルの修正によって改善したりする方がよほど重要と考えます。
    • 単純に「実行処理を分割したい」だけならば、 Task を使った async/await による非同期タスクの仕組みがあるので、 これを使うのが良いでしょう。
    • なお、 Task による非同期タスクは メインスレッド上で実行されます。 マルチスレッド動作にはなりません。

  • Q. 画像を描画したい

  • A. Renderer 名前空間および Texture 名前空間のユーティリティを使うのが良いです。

    • 画像を描画するには、まずVaNillaエンジンに画像を登録し、読み込む処理が必要です。
    • 画像を登録するには、VaNillaEngineManager/CodeGeneratorを用います。
    • ランタイムでは、画像を読み込み、登録するための TextureResourceManagerというユーティリティクラスがあるので、これを使いましょう。
    • 読み込みが完了し、VaNillaエンジン上で扱えるようになった画像は「テクスチャリソース」と呼ばれます。
    • その後、テクスチャリソースを描画する位置や回転角度、拡大率などを Renderer.Utilsモジュールによって指定することで、画面上に描画されます。

  • Q. 現在時刻を取得したい

  • A. Object.Execute() の引数として現在の時刻が渡ってきます。

    • また、 TimeManager.GetCurrentTime() API経由でも時刻の取得ができます。
  • Q. 時刻に基づいた処理を行いたい(例：「一定時間の間無敵状態になる」など)
  • A. TimeSpan を用いるか、 AsyncRunner を使った非同期タスク が使いやすいです。
  • Q. ExpresserComponentの実行優先度を管理したい
  • A. ExpresserComponentExecutionOrderAttribute 属性 を 対象の ExpresserComponent に付与することで制御可能です。
